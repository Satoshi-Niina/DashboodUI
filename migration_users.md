# ユーザー管理 ＆ 認証機能のテナントDB完全移設 報告書 (migration_users.md)

本ドキュメントは、「TownManagerシステム（Dashboard UI）」におけるログイン認証および各種セッション、JWT解決、認証制限ミドルウェアの接続先を共同の `common_db` から、ユーザーが属する各テナント（企業・地方自治体）の個別データベース（`kosei_db`, `daitetsu_db`, `demo_db` 等）へ完全に移行した内容をまとめた計画 ＆ 構築記録書です。

---

## 1. 新仕様のアーキテクチャ概要

ユーザー管理と認証を司令塔（`common_db`）から切り離し、各テナントデータベース直下の `users` 実体テーブルへと完全移入させました。

### 認証処理の流れ
1. **テナントの解決＆決定 (URL/Header検出)**:
   URLパス（例: `/kosei/`）や HTTP ヘッダー `X-Tenant-Id` からテナントIDを特定し、`common_db.public.company_db_routing` 内に登録されている物理的な個別接続データベース（例: `kosei_db`）およびプールを特定します。
2. **非同期実行コンテキストのアクティブバインド**:
   Node.js の `AsyncLocalStorage` により、ログインエンドポイントを呼び出したスレッドの実行プールを、対象のテナントプール（例: `kosei_db`）へ確実にバインドします。
3. **論理リソース引き当て (app_resource_routingによる解決)**:
   テナントDB内の `public.app_resource_routing` 案内表を参照し、論理名 `"users"` から、物理テーブル名 `master_data."users"`（現場データ含む本物の全ユーザーが定義された実スキーマ）を動的に解決。
4. **個別テナントDBでのユーザー整合性・パスワード照合**:
   解決された実テーブルからユーザー情報を取得。これによって、**大鉄環境なら daitetsu_db。近鉄環境なら kintetsu_db 内の users テーブルだけからユーザーを照合、およびパスワード比較（Bcrypt等）をおこなう強固な構造、テナントDB間でのログイン情報の隔離**を保証します。
5. **認証情報の共通化 (JWT生成)**:
   認証成功時には、各個別DBから引き当てた該当ユーザーを一意にするデータおよび外部ロール変換（`mapRoleForExternal`）結果を含む安全性の高い JWT トークン（有効期限 4h）を発行。
6. **リクエスト認可 (要求処理段階でのミドルウェア検証)**:
   今後の各APIへの各種要求（CUD操作等）に対しても、渡されたトークンを引き戻し、対象テナントDBの実テーブルからユーザーデータを引いて管理者であることを確証します。

## 3. `common_db.public.users` の位置づけについて

* **非破壊・形骸状態化の実施**:
  `common_db.public.users`（および `common_db.master_data.users`）テーブルの定義、格納されているデータ、およびテーブルスキーマ自体は一切削除せず、データベース上に残しています。
  
* **使用停止のアライメント**:
  上記修正箇所の通り、ダッシュボードおよび外部連携システムが動作するすべての認証ライフサイクル（ログイン・SSO中継・ verify-token 認証チェック・ユーザーリスト操作の全て）において、`users` に対するクエリは `db-gateway` ルーティング機構、あるいは AsyncLocalStorage 接続スイッチにより**各テナントデータベース（`kosei_db`, `daitetsu_db`等）の `master_data.users` へ100%接続が遷移する状態**を構築しました。
  これにより、司令塔である `common_db.public.users` へのクエリ発行を皆無にし、意図的に使用されない状態を実現しました。

## 4. 影響範囲の検証および動作健全性

* **既存機能（運行計画や検修一覧）の保持**:
  DBプールとスレッドの動的切替ミドルウェアを精密に補強しただけですので、既存の各種DB問い合わせは一切破壊されていません。
* **ローカル・USB環境動作保証**:
  環境変数やローカルなPostgreSQLポート、ローカル開発時特有のフォールバック動作は従前のまま保持されます。
* **URLによるテナント接続先の動的切替**:
  URLパス（/daitetsu等）を介したテナントごとの個別セカンダリデータベースとの統合認証が可能になっています。
